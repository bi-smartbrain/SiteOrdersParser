# Модель данных платформы (brain-сайты + фриланс)

Справочник по сущностям, endpoint'ам и терминологии, которые трогает SiteOrdersParser.
Актуально на 2026-06-18, после релиза 2026-06-15, в котором платформа объединила
«заявку» и «проект» в единую сущность Project на уровне БД, но в UI термины
«заявка» и «проект» остались — за ними теперь стоят разные API-сущности.

## Сайты

Все 5 сайтов работают на единой кодовой базе и имеют одинаковую структуру API:

| Сайт | Назначение |
|---|---|
| rubrain.com | основной |
| junbrain.com | для junior-специалистов |
| engibrain.com | инженерное направление |
| freelance.kz | фриланс-биржа в Казахстане |
| free.uz | фриланс-биржа в Узбекистане |

Для freelance.kz и free.uz токен авторизации нужно получать на их собственном
auth-эндпоинте (`/api/auth/login/`) — токен от rubrain.com к ним не подходит.

## Сущности в API (БД)

Реально в БД живут две связанные сущности:

### Project — заявка от клиента

Создаётся в момент, когда клиент заполнил форму на сайте. Это та запись, по которой
мы шлём уведомления.

- **Список:** `/api/v2/project/manager/control/list/?page=…&size=…`
  - ВСЕ заявки, поступившие на сайт (с любым назначенным менеджером)
  - Для freelance.kz: count = 226 на 17.06.2026
- **«Мои» заявки:** `/api/v2/project/manager/control/mine/list/?page=…&size=…`
  - Только те, что назначены на конкретного бот-юзера как менеджера
  - Для freelance.kz: count = 161
  - **Не используем** — нам нужны все заявки на сайт, а не подмножество
- **Детали:** `/api/v2/project/manager/control/retrieve/{id}/` или `/api/projects/{id}/`
  - В detail дополнительно есть `customer.email`, `customer.phone`, `chat_room`, `briefs`,
    `categories`, `specs`, `stages` (массив опубликованных вакансий)

Ключевые поля Project:

| Поле | Что в нём |
|---|---|
| `id` | внутренний id заявки/проекта |
| `name` | название (часто содержательное, как «Дизайн фото-книги»; иногда автогенеренное «Заявка от 15.06.2026») |
| `created` | ISO 8601 timestamp создания заявки |
| `descr` | основной текст заявки, **приходит с HTML-разметкой** (`<p>...</p>`), на бэке надо стрипать |
| `customer` | подобъект с инфой о клиенте: в списке `{id, first_name, last_name, company}`; в detail добавляются `email`, `phone`, `date_joined`, `role` |
| `manager` | подобъект менеджера, аналогично |
| `manager_status` | строка: `active`, `closed` и т.п. |
| `project_type` | категория проекта (может быть null) |
| `stages_cnt` | сколько Stages (вакансий) создано из этой заявки |
| `environment_created_on` | подобъект с инфо о сайте: `{id, default_lang, group, name}` |

### Stage — вакансия, опубликованная менеджером

Менеджер на основе Project создаёт одну или несколько Stages — это то, что появляется
в публичном каталоге и на что откликаются исполнители.

- **Список:** `/api/v2/stage/manager/control/list/?project=<id>` (опционально фильтр по проекту)
  - Для freelance.kz: count = 145 на 17.06.2026
- В Stage есть `budget`, `currency`, `slug`, `status`, `project` (id связанного Project)

**Связь:** 1 Project → 0..N Stages. Разница (226 − 145 = 81) — это заявки, по которым
менеджер ещё не опубликовал вакансии (либо закрыл как «не подходит»).

## Терминология UI ↔ API

В UI используются «заявка» и «проект», но они означают НЕ то же самое, что одноимённые
сущности на платформе:

| UI термин | UI URL раздела | За ним стоит API-сущность |
|---|---|---|
| **Заявка клиента** | `/account/all-projects/application-list` | **Project** |
| **Проект** (он же вакансия) | `/account/all-projects/vacancies-list` | **Stage** |

На карточке заявки в UI заголовок текста подписан буквально:
**«Описание заявки на проект:»** — это поле `descr` сущности Project.

Карточка одной заявки/проекта в UI: `/account/manager-projects/project/{project_id}`
(c опциональным `?vacancyId=<stage_id>`). Этот URL мы и подставляем в уведомление.

## Endpoint'ы парсера

| Что | Endpoint | Зачем |
|---|---|---|
| Список заявок | `/api/v2/project/manager/control/list/?size=100&page=1` | основной источник истины — все новые заявки на сайт |
| Детали заявки | `/api/v2/project/manager/control/retrieve/{id}/` (или `/api/projects/{id}/`) | подтянуть `customer.email` и `customer.phone`, которых нет в списке |
| Авторизация | `/api/auth/login/?active_lang=ru` | получить access-токен; для freelance.kz и free.uz — на их собственном домене |

## Дедупликация

После релиза 2026-06-15 значения `created` (новый API) и `post_date` (старый API) у одной
и той же физической записи различаются на доли секунды. Поэтому сравнение полей в лоб
не работает.

В коде стоит жёсткий cutoff: `NEW_ENDPOINT_CUTOFF = '2026-06-15T12:00:00+03:00'`.
Всё, что создано до этой отметки, считаем уже записанным; всё, что после —
проверяем по in-memory набору `created`-меток, которые накапливаются за runtime
контейнера.
